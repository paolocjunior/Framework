1. 🏁 O que e o framework
Imagine um colega senior experiente sentado ao seu lado enquanto voce programa. Ele te lembra de planejar antes de codar, revisa seu trabalho, aponta problemas de seguranca, documenta suas decisoes e impede que voce esqueca de rodar testes.
Agora imagine que esse colega nunca fica cansado, nunca esquece nada e funciona 24 horas por dia, em qualquer projeto seu. Esse e o framework.
🎯 O framework transforma o Claude Code de um assistente que &apos;faz codigo&apos; em um parceiro que &apos;entrega software com qualidade senior&apos; — guiado por regras, verificado por camadas e validado por uma segunda IA independente.

2. ⚙️ Instalacao
A instalacao e simples: voce copia a pasta do framework para o seu projeto e esta pronto. O Claude Code detecta automaticamente os arquivos e passa a seguir as regras.
📥 Passo a passo
Passo
Acao
Detalhe
1
📦 Baixe o framework
Clone o repositorio do framework ou baixe o ZIP. Ele contem a pasta .claude/ completa com tudo que voce precisa.
2
📂 Copie 3 itens para o seu projeto
Copie para a raiz do seu projeto: (1) a pasta .claude/ inteira, (2) o arquivo CLAUDE.md e (3) o arquivo AGENTS.md. Esses 3 sao obrigatorios.
3
⚠️ Limpe arquivos residuais
Se veio o arquivo .claude/settings.local.json, DELETE-o (contem permissoes do projeto anterior). Se veio .claude/runtime/.plan-approved, DELETE-o tambem. Esses arquivos sao gerados em tempo de execucao e nao devem ser copiados entre projetos.
4
🔑 De permissao aos scripts
No Linux ou Mac, rode chmod +x .claude/hooks/*.sh para que os scripts possam ser executados. No Windows com Git Bash, o framework ja cuida disso.
5
💻 Abra o Claude Code
Abra o terminal na pasta do projeto e inicie o Claude Code. Ele vai carregar automaticamente as regras e ativar os hooks.
6
✅ Verifique se esta tudo certo
Na primeira mensagem, digite /status-check. Se aparecer um resumo do estado do projeto, deu certo.
🔧 Dependencia importante
⚠️ O framework usa a ferramenta jq para processar dados dos hooks. A maioria dos sistemas ja tem, mas se o framework reclamar, instale com: apt install jq (Linux), brew install jq (Mac) ou baixe do site oficial (Windows).
🌐 Revisor externo (opcional mas recomendado)
Para ativar a segunda IA revisora (Codex/GPT-5.4), siga estes passos:
Instale o Codex CLI: npm install -g @openai/codex
Faca login: codex login
Instale o plugin no Claude Code: /install-plugin openai/codex-plugin-cc
Recarregue: /reload-plugins
Configure o AGENTS.md na raiz do projeto (template incluido no framework)
💡 Sem o Codex você ainda tem o framework completo, mas perde a camada de validação cross-model (Camada 4). O framework preenche o AGENTS.md automaticamente quando o /spec-check dá READY ou READY WITH ASSUMPTIONS.
📡 Sensores — Configuracao inicial
Sensores sao verificacoes mecanicas (testes, lint, build) que o framework executa. Para configura-los:
Copie .claude/runtime/sensors.template.json para .claude/runtime/sensors.json
Edite para refletir a stack real do seu projeto (remova sensores nao aplicaveis)
Rode /sensors-run apenas quando os comandos declarados ja forem executaveis — em projetos sem codigo ainda, execute o baseline apos o bootstrap minimo da stack
Comite sensors.json no repositorio
💡 Projetos sem sensors.json operam em modo degradado — o framework funciona, mas reporta a ausencia como lacuna.

4. 🧭 Os 12 passos do dia a dia
Aqui esta o fluxo completo, passo a passo. Siga na ordem e o framework garante o resto.
📝 Cada passo tem 3 blocos: 👤 o que VOCE faz, 🤖 o que o FRAMEWORK faz, e ✅ como saber que FUNCIONOU. Se voce esta comecando, so siga a coluna VOCE — o resto acontece sozinho.
Passo 1 — 📝 Criar a especificacao
Tudo comeca descrevendo o que voce quer construir. Nao precisa ser tecnico.
Quem
O que
👤 Voce
Digita /spec-create e responde as perguntas sobre o produto que quer construir.
🤖 Framework
Faz perguntas guiadas sobre objetivo, publico, telas, dados, regras de negocio e escopo. No final, gera documento estruturado.
✅ Sinal de OK
Apareceu especificacao completa com requisitos numerados (AUTH-01, TASK-02), telas, modelo de dados e criterios de aceite.
💡 Voce pode comecar com algo simples: &apos;Quero criar um site para controlar financas pessoais.&apos; O framework faz o resto.
Passo 2 — 🔎 Validar a especificacao
Quem
O que
👤 Voce
Digita /spec-check.
🤖 Framework
Lê a spec, procura lacunas, classifica cada problema e dá veredicto: READY, READY WITH ASSUMPTIONS ou BLOCKED.
✅ Sinal de OK
Veredicto READY ou READY WITH ASSUMPTIONS. Se vier BLOCKED, lista exatamente o que precisa ser resolvido antes de implementar.
⚠️ Se der BLOCKED: não tente contornar. Responda às perguntas e rode /spec-check de novo. Cada ambiguidade ignorada vira um bug futuro.
Passo 3 — 🎨 Escolher o visual (se tem UI)
Quem
O que
👤 Voce
Digita /design-preview e depois /ui-plan.
🤖 Framework
Gera opcoes de paleta, tipografia, espacamento, componentes base. Voce escolhe e o framework congela o padrao visual.
✅ Sinal de OK
Design System aprovado. Telas futuras seguem automaticamente essa identidade visual.
Passo 4 — 🗺️ Planejar a implementacao
Quem
O que
👤 Voce
Digita /plan.
🤖 Framework
Executa o pre-check obrigatorio (OIs antes_do_plan e artefatos prometidos pela spec), desenha o plano completo e, quando o Codex esta configurado, roda revisao adversarial antes de apresentar. O que voce ve ja e o plano pos-Codex, com Delta pos-Codex quando houver findings.
✅ Sinal de OK
Plano apresentado no chat. Voce le, aprova ou pede ajustes.
🎯 Mesmo para mudancas simples, o plano e obrigatorio. Essa disciplina evita os erros mais comuns.
Passo 5 — ✅ Revisar o plano
Quem
O que
👤 Voce
Digita /plan-review.
🤖 Framework
Chama dois agentes em paralelo: um compara plano com spec, outro checa coerência interna. Devolve APPROVED, APPROVED_WITH_CORRECTIONS, NEEDS_REVISION ou NEEDS_HUMAN_REVIEW, com findings BLOCKING, NON-BLOCKING ou EDITORIAL. Quando aplicável, inclui APPLICABLE_DELTA para orientar correções diretamente aplicáveis.
✅ Sinal de OK
Plano APPROVED ou APPROVED_WITH_CORRECTIONS. O framework cria marker .plan-approved que libera o próximo passo. NEEDS_REVISION ou NEEDS_HUMAN_REVIEW não liberam implementação.
⚠️ Portao mecanico: um hook BLOQUEIA criacao de codigo enquanto o plano nao for aprovado pelo /plan-review. Nao e sugestao — e impedimento real.
Passo 6 — ⌨️ Implementar o codigo
Quem
O que
👤 Voce
Pede &apos;implemente a Fase 1 do plano&apos; ou equivalente.
🤖 Framework
Escreve codigo arquivo por arquivo. A cada edicao, hooks rodam automaticamente checando sintaxe, secrets, padroes, design tokens, qualidade.
✅ Sinal de OK
Arquivos criados sem alertas dos hooks. Codigo sintaticamente valido e sem problemas obvios.
Passo 7 — 🔍 Revisar o codigo
Quem
O que
👤 Voce
Digita /review.
🤖 Framework
Chama em paralelo: revisor de codigo (qualidade senior), auditor de seguranca (vulnerabilidades) e auditor de QA (cobertura de testes).
✅ Sinal de OK
Review publicado com findings classificados por gravidade (CRITICO, ALTO, MEDIO, BAIXO).
Passo 8 — 🛡️ Auditoria profunda (quando aplicavel)
Comando
Foco
/audit
Auditoria geral de seguranca e qualidade
/db-audit
Foco em banco de dados: queries, indices, seguranca, integridade
/web-audit
Foco em APIs web: autenticacao, autorizacao, XSS, CSRF, CORS
/k8s-audit
Foco em Kubernetes: privilegios, network policies, secrets, RBAC
💡 Nao precisa rodar todas. Use /audit como padrao. Adicione as variantes conforme o projeto.
Passo 9 — 🩹 Corrigir o que apareceu
Quem
O que
👤 Voce
Pede ao Claude Code para corrigir os findings, comecando pelos mais graves.
🤖 Framework
Aplica os fixes. Cada correcao passa pelos hooks novamente. Depois do fix, busca o mesmo padrao em TODO o projeto para evitar fix parcial.
✅ Sinal de OK
Todos os CRITICOS e ALTOS corrigidos. Re-rodar /review confirma que os findings foram resolvidos.
Passo 10 — 📝 Documentar as decisoes
Quem
O que
👤 Voce
Digita /justify.
🤖 Framework
Gera bloco formatado com as justificativas tecnicas: decisao, alternativas rejeitadas, motivo.
✅ Sinal de OK
Decisoes importantes viraram patrimonio do projeto.
Passo 11 — 🎯 Verificar contra a especificacao
Quem
O que
👤 Voce
Digita /verify-spec.
🤖 Framework
Compara cada requisito da spec com o código. Marca MATCH, PARCIAL ou MISS. Usa sensores, behaviours e contratos como evidência mecânica. Para tokens marcados como [literal], grep é apenas sinal inicial: a conformidade exige binding rastreável ao comportamento verificado.
✅ Sinal de OK
Todos os requisitos v1 marcados como MATCH.
Passo 12 — 🚀 Checagem final de entrega
Quem
O que
👤 Voce
Digita /ship-check.
🤖 Framework
Checa: build, testes, lint, secrets, config, dependencias, observabilidade, performance. Valida sensores, contratos, behaviours, linters, KB e gaps. Chama risco final.
✅ Sinal de OK
Veredicto PRONTO. Se PRONTO COM RESSALVAS, voce decide. Se NAO PRONTO, corrige antes.
🎉 Chegou no veredicto PRONTO? Esta pronto para distribuir! O framework garantiu que o projeto passou por todas as camadas de qualidade.
5. 🤖 O que acontece automaticamente
Enquanto você trabalha, o framework tem 12 scripts rodando automaticamente. Você não precisa chamar nenhum deles — eles se ativam sozinhos no momento certo.
⏰ Quando cada um dispara
Momento
Quando dispara
O que acontece
🚀 Inicio da sessao
Quando abre ou retoma o Claude Code

🛡️ Antes de cada edicao
Quando o Claude vai modificar arquivo
Protect-files bloqueia arquivos sensiveis e pre-implementation-gate exige plan-review
✅ Depois de cada edicao

No encerramento da sessao

🔍 O que eles protegem contra
Problema
O que acontece
🔑
Secrets hardcoded
API keys, senhas, tokens colados direto no codigo
💀
Codigo perigoso
eval(), exec(), Function() e execucoes dinamicas
🎨
Cores soltas
Hex codes misturados no lugar do design system
🎲
Nao-determinismo
Math.random() e new Date() onde deveria ser fixo
🔒
Arquivos sensiveis
Edicoes em .git/, lockfiles, arquivos de build
📋
Pulos de portao

🚧
Sintaxe quebrada
Python que não compila, erros óbvios
🚫
.gitignore incompleto
Caches, builds, ambientes virtuais e artefatos de stack que poderiam ser commitados por engano
💡 Você não precisa lembrar de nada. Todos esses checks são automáticos. Você só vai perceber quando algo der problema — e aí o framework mostra exatamente o quê, onde e como resolver.

10. 📡 Camadas avancadas
Alem do workflow basico (spec → plan → implement → review → ship), o framework oferece 7 camadas avancadas OPT-IN que adicionam verificacao mecanica profunda.

🏃 Sprint Contracts
🎯 Unidades ATOMICAS de entrega (1-2h) com evaluator deterministico. Phase = dias/semanas. Sprint = horas.
⚠️ Invariante central: phase contract NUNCA e mutado por sprint. Vinculo e via filesystem.

Verdict vem de
file_exists
path existe no filesystem
grep_pattern
matches de regex em arquivo
sensor_subset
sensors-last-run.json
custom_command
exit code (timeout obrigatorio)
🧪 Behaviour Harness
🎯 Behaviours DISPARAM acao real e COMPARAM resultado vs expectativa. Sensores = &apos;compila?&apos;. Behaviours = &apos;quando executo X, FAZ Y?&apos;
Tipo de Expectation
O que compara
exit_code
Exit code do comando bate com esperado
stdout_contains
stdout contem pattern esperado
stdout_json_path
filtro jq no stdout JSON bate
file_content
arquivo contem pattern esperado
file_exists_after
arquivo existe apos execucao
not_contains
stdout NAO contem pattern (negativa)
📝 Binding bidirecional: behaviour declara contract_ref e phase contract declara behaviour_id. Ambas as pontas devem existir.

12. 🎬 Cenarios praticos
Situacoes reais que voce vai encontrar no dia a dia e como o framework te ajuda.
🎯 Cenario 1 — &apos;Quero comecar um projeto novo&apos;
🎬 Voce tem uma ideia e quer comecar do zero.
Rode /spec-create e converse com o framework sobre o produto.
Quando terminar, rode /spec-check. Se der BLOCKED, resolva e tente de novo.
Com READY, rode /design-preview se tem interface. Escolha uma opcao.
Rode /plan para gerar o plano de implementacao.
Rode /plan-review. Se vier APPROVED ou APPROVED_WITH_CORRECTIONS, está liberado para implementar.
Peca ao Claude Code para implementar a Fase 1. Ele escreve; os hooks vigiam.
Ao final da fase, rode /review para revisao.
Corrija findings, documente com /justify, e prossiga para a proxima fase.
💡 Uma spec completa de projeto pequeno leva 20-40 minutos de conversa. O plano, 5-10 minutos.
🔧 Cenario 2 — &apos;Preciso corrigir um bug&apos;
🎬 Alguem reportou que a funcao X esta errada.
Rode /status-check para confirmar o estado atual.
Rode /plan descrevendo o bug e o fix pretendido.
Rode /plan-review.
Com APPROVED ou APPROVED_WITH_CORRECTIONS, peça a correção ao Claude Code.
O framework aplica o fix e busca o mesmo padrao em TODO o projeto.
Rode /review para confirmar e /verify-spec se o bug afetava requisito da spec.
🎯 Nenhum fix e &apos;pequeno demais&apos;. Mesmo para um typo, o framework pede o fluxo completo.
🚀 Cenario 3 — &apos;Quero fazer o primeiro deploy&apos;
🎬 O projeto esta implementado e voce vai colocar no ar.
Rode /verify-spec para confirmar que todos os requisitos foram entregues.
Rode /audit (e as variantes que fazem sentido).
Corrija tudo que for CRITICO e ALTO.
Rode /ship-check. Esse e o ultimo portao.
Se PRONTO, pode entregar. Se COM RESSALVAS, decida. Se NAO PRONTO, resolva.
🔄 Cenario 4 — &apos;Voltei depois de 2 semanas sem mexer&apos;
🎬 Voce abandonou o projeto. Como retomar sem perder contexto?
Abra o Claude Code na pasta do projeto.
O framework carrega automaticamente a memoria: estado, pendencias, decisoes.
Rode /status-check para ver o resumo.
Se tiver muita coisa acumulada, rode /memory-consolidate.
Continue de onde parou.
💡 A memoria do projeto (estado, decisoes, findings, pendencias) nao depende da sessao. Quando voce volta, o framework reconstroi o contexto a partir dos arquivos — nao importa quanto tempo passou.
13. ❓ Perguntas frequentes
❓ Preciso saber programar para usar o framework?
Para tirar proveito completo, sim. Mas para acompanhar e entender o que esta acontecendo, nao. O framework explica cada passo, e voce pode pedir ao Claude Code para explicar qualquer coisa.
❓ Posso usar em qualquer linguagem de programacao?
Sim. O framework é genérico. Alguns hooks têm checks específicos por stack, e o gitignore-guard detecta stacks comuns como Node, Python, Rust, Go e Java. A camada de regras, comandos e agentes continua genérica e funciona em qualquer stack.
❓ E se eu pular um passo?
Alguns passos tem portoes mecanicos (como o /plan-review antes de implementar). Esses nao da para pular — o framework bloqueia. Outros sao disciplinares: tecnicamente da, mas voce perde as garantias de qualidade.
❓ O framework funciona offline?
Os hooks e comandos sim. A parte de cross-review com Codex precisa de conexao. O Claude Code em si tambem precisa de conexao (chama a API da Anthropic).
❓ Posso customizar as regras?
Pode. As regras vivem em arquivos .md dentro de .claude/rules/. Voce pode editar, adicionar novas, remover as que nao fazem sentido. O framework e seu — adapte como precisar.
❓ Como o framework se compara a rodar o Claude Code sem ele?
Sem framework: voce depende da memoria do assistente e da disciplina de pedir revisao. Com framework: tudo e estruturado, automatizado, verificado em multiplas camadas e guardado entre sessoes. A diferenca aparece em projetos medios para grandes.
❓ Quanto o framework adiciona de tempo ao trabalho?
No comeco, parece adicionar. Depois do segundo ou terceiro projeto, voce percebe que economiza tempo — porque os bugs que voce teria nao acontecem, e as decisoes que voce teria esquecido estao documentadas.
❓ Posso usar em projetos profissionais?
O framework foi desenhado exatamente para projetos profissionais. Use em projetos pessoais para aprender o fluxo, depois leve para o trabalho.
❓ Preciso atualizar o framework com frequencia?
Nao. O framework e estavel. Quando sair versao nova com melhorias (como V4+), basta copiar os arquivos atualizados para o seu projeto.
❓ E se o framework reclamar de algo que eu sei que esta certo?
Leia a justificativa que o framework apresenta. As vezes ele esta certo e voce nao percebeu. Se depois de ler voce ainda achar que ele esta errado, pode discordar e seguir — o framework avisa, nao obriga (exceto nos portoes mecanicos).
❓ O que sao as &apos;camadas avancadas&apos; (sensores, contratos, behaviours...)?
Sao ferramentas OPT-IN para projetos que querem verificacao profunda. O framework basico funciona sem elas. Conforme o projeto cresce, voce pode adotar sensores (verificacao mecanica), contratos (compromisso formal por fase), behaviours (teste de comportamento real), linters (invariantes estruturais), knowledge base (documentacao sintetizada) e gaps (rastreamento de lacunas).
